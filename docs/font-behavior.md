# 字体行为与浏览器回退

UnicodeArtJs 会把“输入文字如何被渲染成采样图像”和“字符画最终如何显示”分成两类字体。理解这两个概念，可以减少“换了字体但看起来没变化”或“裱框对齐不一致”的误判。

## 两类字体

- **视觉字体**：用于文字 Banner 模式，把输入文字渲染成中间图像，再由 Core 转成字符画。它会影响字形轮廓、笔画粗细和采样结果。
- **字素字体**：用于字符画的字素模板匹配、预览、HTML 导出和 PNG 导出。它决定最终字符画在网页、编辑器或终端中的对齐效果。

建议使用开源等宽或混合等宽字体，例如 Sarasa Mono SC、LXGW WenKai Mono、Source Code Pro、Liberation Mono。中文字符画尤其依赖“ASCII 宽度一致，中文和部分 Unicode 字符严格为两倍宽”的混合等宽行为。

## 为什么字体可能不生效

Web 工具站不打包字体文件，也不会从第三方地址自动下载字体。浏览器只会使用访问者设备上已安装且当前站点可访问的字体；如果字体缺失，浏览器会按 `font-family` 列表自动回退。

常见原因：

- 本机未安装所选字体，或字体名称与浏览器识别的 family name 不一致。
- 浏览器隐私保护、字体指纹保护或扩展限制了本机字体检测。
- 视觉字体只影响文字 Banner 的采样，不会改变预览区显示字体。
- 字素字体需要重新生成字符画，因为它也参与字素模板匹配。
- 终端、网页和编辑器使用的实际字体不同，同一段字符画会呈现出不同宽度。

Web 页面会显示字体可用性提示。这个提示使用 Canvas 量测判断目标字体是否明显区别于通用 fallback，只用于帮助诊断，不是严格许可证、安装或排版验证工具。

## Brave 与隐私保护模式

Brave 等浏览器可能启用字体指纹保护。启用后，页面仍会设置你选择的 `font-family`，但浏览器可能对本机字体访问、检测或回退策略做额外限制。表现可能包括：

- 下拉框值已经变化，但预览区看起来仍像默认字体。
- 字体可用性提示显示“未确认”，即使系统里已经安装字体。
- 同一页面在 Chrome 或 Edge 正常，在 Brave 中不明显变化。

处理方式：

1. 先确认本机确实安装了字体，并在其它应用中能看到同名 family。
2. 在同一浏览器中刷新页面并重新生成字符画。
3. 若使用 Brave，检查当前站点的 Shields 或字体指纹保护设置。
4. 需要稳定对比时，优先使用 Chrome 120+ 或 Edge 作为基线浏览器。

## 网络字体与宿主集成

官方 GitHub Pages 工具站暂不提供“输入任意网络字体 URL 并加载”的功能，原因是它会带来 CORS、缓存、字体许可证、远程资源可用性和隐私提示等额外边界。

二次开发者或自建站点可以在自己的宿主页面中显式加载受许可的字体，例如：

```css
@font-face {
  font-family: "My Open Font";
  src: url("/fonts/my-open-font.woff2") format("woff2");
  font-display: swap;
}
```

然后在 Core 配置中传入同名字体：

```js
const result = await textToArt("字体", {
  visualFont: { family: '"My Open Font", sans-serif' },
  glyphFont: { family: '"My Open Font", monospace' },
  outputTarget: 'web'
});
```

自建页面应明确字体来源和许可证，并确保字体文件由自己有权分发的域名提供。未来如果 UnicodeArtJs 官方工具站提供自定义网络字体位置，也会优先采用“用户明确输入、明确授权、同源或受控域名、只保存在本地浏览器”的模式。
